# Learning Content

When the user asks for tutorials, learning guides, study notes, technical blog posts, or content meant to help a developer level up, do not produce a shallow checklist or generic theory dump.

Build the content like a real engineering walkthrough:

1. Start from the practical problem and the expected outcome.
2. Show the concrete input/output or user-facing behavior.
3. Derive requirements from the outcome, including non-functional constraints.
4. Compare meaningful implementation approaches and their trade-offs.
5. Recommend one approach and explain why it fits the current project stage.
6. Define the core concepts, models, interfaces, abstract classes, or concrete classes needed to solve the problem.
7. Explain how those pieces collaborate end-to-end until the expected result is produced.
8. Include code snippets at the important decision points, not only prose.
9. Include package/folder structure when it clarifies boundaries or implementation flow.
10. Cover test strategy, acceptance criteria, and realistic extension paths.

Prefer fewer, deeper sections over many repetitive short files. Merge overlapping theory. Avoid generic "senior mindset" summaries unless they are tied to a concrete design decision.

For Vietnamese learning content, write in natural Vietnamese with diacritics. Keep the tone practical and mentor-like: explain the reasoning, show the code, name the trade-off, then connect it back to what the reader can reuse in future work.
